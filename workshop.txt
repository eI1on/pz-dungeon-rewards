title=[B41] Dungeon Rewards
description=[h1]Dungeon Rewards[/h1]
description=
description=Turn [b]any container[/b] in the world into a server-authoritative [b]dungeon chest[/b]: admins build [b]weighted reward presets[/b], bind them to specific containers, and players open a polished [b]card-reveal roll[/b] before claiming loot. Built for event maps, quest hubs, and community servers that want loot without hand-placing unique items for every run. Admins configure presets and bindings through the [b]Dock Menu[/b] panel ([b]ElyonLib[/b]); players still interact by opening registered containers in the world.
description=
description=[h1]How players use it[/h1]
description=[list]
description=[*]When a container is registered and assigned a preset, use the context menu [b]Open[/b] on that object to launch the player window.
description=[*]Your [b]roll is generated on the server[/b] and shown as flip/reveal cards ([i]items show icons where available[/i]).
description=[*][b]Claim[/b] applies the rolled bundle; if the preset is [i]once per player[/i], the container remembers so you cannot bypass it client-side.
description=[*]Already claimed containers surface a clear status instead of re-rolling silently.
description=[/list]
description=
description=[h1]How admins set it up[/h1]
description=[list]
description=[*]Right-click a container with admin access: [b]Register[/b] it into Dungeon Rewards, then open the [b]Admin[/b] panel to assign a [b]preset[/b], rename entries, or toggle whether the container is active.
description=[*]Design presets entirely in-game: reward rows, weights, [b]roll count[/b] (how many weighted picks fire per open), [b]allow duplicates[/b], and [b]consume once per player[/b] vs repeatable chests.
description=[*][b]Import[/b] / [b]export[/b] the JSON blueprint for backups, sharing setups between worlds, or editing large tables offline ([i]see [b]Data files[/b] below[/i]).
description=[*][b]Reset container claims[/b] from the admin tools when you want the same physical chest to pay out again after a dungeon reset or seasonal event.
description=[/list]
description=
description=[h1]Reward types[/h1]
description=Each preset row can mix any of the following; [b]granting happens on the server[/b] when the player confirms the claim:
description=[list]
description=[*][b]Items[/b] - type plus stack count ([i]standard identifiers such as [/i][b]Base.Axe[/b][i].[/i]).
description=[*][b]XP[/b] - perk or skill plus positive or negative amounts.
description=[*][b]Traits[/b] - add or remove a trait when valid; skips impossible combinations using the same ElyonLib reward-grant pipeline as items and XP.
description=[*][b]Custom Lua[/b] - register handlers with [i]RegisterCustomReward[/i] from another mod to run scripted payouts ([i]ideal for currency, teleport tokens, or dungeon keys[/i]).
description=[/list]
description=
description=[h1]Roll fairness notes[/h1]
description=The host keeps each player's [b]pending roll[/b] until it is claimed or cleared. When many players open the same container, the service [b]attempts to avoid repeating the exact same roll signature[/b] back-to-back so coordinated groups see a little more variety ([i]still weighted - very small pools can collide[/i]).
description=
description=[h1]Data files[/h1]
description=Persistence is written under your usual Zomboid Lua data layout for the [i]Dungeon Rewards[/i] mod context:
description=[list]
description=[*]authored preset library saved by the server:
description=[*][code]presets.json[/code]
description=[*]registered world containers and which preset they use:
description=[*][code]containers.json[/code]
description=[*]per-player claim flags and pending rolls
description=[*][code]player_rolls.json[/code]
description=[*]portable import/export bundle for sharing configurations
description=[*][code]presets_export.json[/code]
description=[/list]
description=Prefer the in-game importer after manual edits so malformed JSON is caught early.
description=
description=[h1]Requirements[/h1]
description=[list]
description=[*][b]ElyonLib[/b] - load it before Dungeon Rewards. It exposes the Dock Menu admin panel, the themed UI, server/client networking for rolls and claims, and the reward pipeline (items, XP, traits, custom hooks).
description=[/list]
description=
description=[h1]Singleplayer & multiplayer[/h1]
description=[list]
description=[*][b]Dedicated / hosted servers:[/b] presets, containers, and player rolls stay authoritative on the host.
description=[*][b]Listen / co-op host:[/b] same logic - whoever runs the session server owns the data.
description=[*][b]Singleplayer:[/b] fully supported; your local save is simply both client and host.
description=[/list]
description=
description=[h1]Frequently asked questions[/h1]
description=
description=[h2]Does it need a new save?[/h2]
description=No - add it when you like; JSON files populate on first run.
description=
description=[h2]Can players cheat extra rolls?[/h2]
description=No - rolls, claim checks, and inventory grants are enforced server-side; the UI only displays what the host already decided.
